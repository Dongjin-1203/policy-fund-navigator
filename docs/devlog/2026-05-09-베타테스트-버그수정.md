## 2. 인프라 변경

### 2-1. EC2 t3.large 업그레이드 + SSM Session Manager 전환

베타 테스트 전 t3.medium에서 t3.large(RAM 8GB)로 업그레이드했다. ChromaDB 임베딩 모델(KR-SBERT) 로드 시 OOM이 발생했기 때문이다. 동시에 SSH 포트 개방 대신 AWS SSM Session Manager로 접속 방식을 전환했다. 보안 그룹에서 22번 포트를 제거하고 IAM 역할에 `AmazonSSMManagedInstanceCore` 정책을 부여하는 것만으로 키 파일 없이 브라우저 콘솔에서 셸 접근이 가능해졌다.

### 2-2. NEXT_PUBLIC_API_URL https:// 누락 수정

프론트엔드가 API를 찾지 못하고 404를 반환했다. 원인은 `.env`의 `NEXT_PUBLIC_API_URL` 값에 `https://` 프리픽스가 없어 상대 경로로 해석된 것이었다. 배포 환경에서 환경변수 값 자체를 검사하는 것이 디버깅 첫 단계여야 한다는 것을 다시 확인했다.

### 2-3. Airflow CeleryExecutor Worker 미기동 수정

`airflow-worker` 서비스가 healthcheck 실패로 재시작 루프에 빠져 DAG가 `queued` 상태에서 진행되지 않았다. `celery[redis]` 패키지 버전 충돌이 원인이었다. `requirements.txt`에 `celery==5.3.6`을 고정하고 이미지를 재빌드했다.

### 2-4. AWS SignatureDoesNotMatch (Secret Key 중복) 수정

S3 API 호출 시 `SignatureDoesNotMatch` 에러가 발생했다. `.env`에 `AWS_SECRET_ACCESS_KEY`가 두 개의 키 이름(`AWS_SECRET_KEY`, `AWS_SECRET_ACCESS_KEY`)으로 중복 설정되어 있었고 boto3가 잘못된 값을 참조했다. 하나로 통일했다.

## 3. 주요 버그 수정

### 3-1. program_features.parquet 핵심 필드 100% null

**증상**: API 응답에서 `max_support`, `interest_rate`, `industry_limit`, `debt_ratio_limit`, `requirements` 모두 null.

**원인 (복합)**:

① **JOIN 키 불일치**: `merge.py`가 `filename` 기준으로 requirements_db와 JOIN하려 했으나 requirements_db 파일명은 `{slno}_{idx}.json` 포맷이어서 매칭 실패.  
② **flat/nested 구조 혼재**: requirements_db에 `{"programs": [...]}` (nested) 형태와 `{"program_name": ..., "source_file": slno}` (flat) 형태가 공존해 `get('programs') or []`가 flat 파일에서 `[]`를 반환.  
③ **필드명 불일치**: 구 포맷 `sub_title`/`program_category`와 신 포맷 `program_name`/`category`를 구분하지 않아 모두 null.

**수정**:
- `_load_all_requirements()`: flat 파일을 `source_file` 또는 `{stem}의 _N 제거`로 slno를 추출해 `flat_buffer`에 묶어 nested 포맷으로 래핑. nested 포맷이 있으면 우선 적용.
- `_build_rows()`: `prog.get('program_name') or prog.get('sub_title')` 등 이중 필드명 폴백 적용.

### 3-2. program_id 포맷 불일치 → matched_count: 0

**증상**: ChromaDB `search_for_agent()`에서 89건이 통과(`검증 완료 - 통과: 89`)했으나 최종 `matched_count: 0`.

**원인**: 교집합(intersection) 체크에서 ChromaDB `program_id`와 parquet `program_id` 포맷이 달랐다.

| 위치 | 포맷 | 예시 |
|---|---|---|
| requirements_db JSON | `{slno}_{i}` (0-indexed) | `633_0` |
| ChromaDB | requirements_db 그대로 | `633_0` |
| parquet (수정 전) | 단일: `{slno}`, 복수: `{slno}_{i+1}` | `633` 또는 `633_1` |

`embedding_node`의 교집합 루프(`pid not in hard_passed_ids`)가 항상 True → 모든 후보 스킵.

**수정**: `_build_rows()`에서 `pid = f'{slno}_{i}'` (0-indexed)로 통일.

### 3-3. interest_rate Optional[float] → Optional[str]

금리 문자열(`"정책자금 기준금리에서 0.15%p 차감"`)을 Pydantic이 `float`으로 변환하려다 null 처리. `ProgramItem.interest_rate`를 `Optional[str]`로 변경하고 `field_validator`로 `'none'/'nan'/''` → `None` 처리를 추가했다.

### 3-4. apply_end "nan"/"None" 문자열 처리

`_clean_apply_end()`를 `match.py`에 추가해 `'None'`, `'nan'`, `''`을 모두 `None`으로 정규화했다. embedder.py의 `_hard_filter()`에서도 `strptime` 전에 명시적 유효성 체크를 추가해 `ERROR 날짜 포맷 오류` 로그가 해소됐다. (`예산 소진시까지`, `세부사업별 상이` 등의 텍스트는 파싱 불가이므로 `apply_end = None`으로 처리.)

### 3-5. is_venture/is_innobiz G점수 미반영

**원인**: 프론트엔드 폼이 `is_venture: true`를 `financial_data` 내부에 전송했으나 `FinancialData` 스키마에 해당 필드가 없어 Pydantic이 무시. `company_features["is_venture"]`는 `req.is_venture`(기본값 `False`)만 읽어 G점수 항상 0.

**수정**: `FinancialData`에 `is_venture`, `is_innobiz` 추가. `_build_initial_state()`에서 `fd.is_venture is not None`이면 최상위 값을 덮어씀.

### 3-6. `float('nan')` is truthy → `'float' object is not iterable`

```python
# 수정 전 — NaN은 truthy이므로 nan or [] = nan
industry_limit = prog.get('industry_limit') or []
for limit in industry_limit:  # TypeError: 'float' object is not iterable
```

```python
# 수정 후
industry_limit = prog.get('industry_limit')
if isinstance(industry_limit, str):
    ...
elif not isinstance(industry_limit, list):
    industry_limit = []  # NaN, None, float 등 처리
```

Python에서 `float('nan')`은 `bool(nan) == True`이므로 `or []` 패턴이 nan을 그대로 통과시킨다. 이진 로직으로 방어해야 한다.

### 3-7. Gemini 피드백 중복 10건

**원인**: `_phase_generate_feedback()`이 `ranked_programs`를 dedup 없이 Gemini 프롬프트와 프로그램 목록에 직접 사용. `match.py`의 dedup은 API 응답 변환 시점에만 수행됨.

**수정**: `_dedup_ranked(program_id 기준)` 헬퍼를 추가하고 `_phase_generate_feedback()` 진입 직후 호출.

### 3-8. ChromaDB 중복 문서 초기화 후 재인덱싱

`PolicyVectorStore.add_documents()`가 중복 체크 없이 호출되어 동일 `program_id` 문서가 여러 번 적재됐다. `reset=True`로 ChromaDB를 초기화한 뒤 재인덱싱해 89 → 100건 (중복 없음) 으로 정리됐다.

## 4. T점수 가중치 복원 및 S3 KIPRIS 폴백

### 4-1. 가중치 복원

```python
# 수정 전 (임시값)
_T_PATENT_WEIGHT  = 1.0
_T_IPC_WEIGHT     = 0.0
_T_RECENCY_WEIGHT = 0.0

# 수정 후 (설계 의도 반영)
_T_PATENT_WEIGHT  = 0.6
_T_IPC_WEIGHT     = 0.2
_T_RECENCY_WEIGHT = 0.2
```

### 4-2. 동적 가중치 재배분

`ipc_codes`, `latest_patent_year`가 없으면 해당 가중치를 `patent_score`에 자동 재배분. KIPRIS 연동 전에는 사실상 `patent_count`가 1.0 가중치를 가져가고, 연동 후에는 자동으로 3분할 적용됨.

### 4-3. S3 KIPRIS 폴백

`patent_count`가 `company_features`에 없을 때 `corp_name` 기반으로 `raw/kipris/{date}/{corp_name}.json`에서 최신 데이터를 조회하는 폴백 로직을 `_fetch_patent_count_from_s3()`으로 구현.

## 10. 배운 점

**교집합 체크의 id 포맷은 모든 저장소에서 동일해야 한다.** ChromaDB, S3 JSON, parquet 세 곳이 각자 다른 `program_id` 규칙을 쓰면 교집합은 항상 공집합이 된다. 오늘 `{slno}_{i+1}` vs `{slno}_{i}` 1씩 차이가 전체 매칭 파이프라인을 무력화했다. 공유 식별자는 생성 시점에 단일 컨벤션으로 고정하고 코드에 상수로 정의해야 한다.

**로그가 없으면 버그가 숨는다.** ChromaDB `통과: 89` 로그와 `matched_count: 0` 사이의 간극은 `intersection check: overlap=0` 로그 한 줄로 즉시 드러났다. 파이프라인 각 단계의 입출력 건수를 INFO 레벨로 찍는 것이 디버깅 시간을 수십 배 줄인다.

**Pydantic은 선언되지 않은 필드를 조용히 무시한다.** `is_venture`를 `financial_data` 안에 넣어도 스키마에 없으면 에러 없이 드롭된다. 스키마 불일치는 런타임 에러를 일으키지 않아 찾기 어렵다. 클라이언트-서버 간 페이로드 계약은 단위 테스트로 검증해야 한다.
