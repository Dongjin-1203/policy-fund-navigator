# project-onboarding 스킬

## 목적
환경이 바뀌거나 며칠 후 작업을 재개할 때,
프로젝트 전체 컨텍스트를 빠르게 파악하고
즉시 작업을 이어갈 수 있도록 표준화된
온보딩 절차를 제공한다.

## 트리거 조건
아래 상황에서 반드시 이 스킬을 적용할 것:
- "작업 재개", "이어서 진행", "컨텍스트 파악" 언급 시
- 환경 전환 (PC 변경, 노트북 → 데스크탑 등) 언급 시
- 며칠 만에 돌아왔다는 언급 시
- 새 세션에서 프로젝트 작업 시작 시
- "현재 상황 정리해줘", "어디까지 했지" 언급 시

## 실행 순서

### Phase 1. 문서 파악

1. 아래 순서로 존재하는 파일을 읽을 것.
   없는 파일은 스킵하고 "파일 없음"으로 표기.
   - CLAUDE.md 또는 README.md
   - PRD, 기획서, 설계문서 등 핵심 문서

2. 회고/개발일지 파악

   [회고/개발일지 있는 경우]
   docs/devlog/ 에서 가장 최근 파일 1개를 읽고
   "남은 작업" 또는 "다음 스텝" 섹션 파악.
   파일명(날짜)으로 마지막 작업 시점도 확인.

   [회고/개발일지 없는 경우]
   git log --oneline -10 기반으로
   최근 작업 내용을 추정하여 요약.
   "회고 없음 — git log 기반 추정"으로 표기.

### Phase 2. 최신 코드 동기화 및 브랜치 상태 파악

3. 최신 코드 동기화

   ① git fetch --all
      실패 시 "fetch 실패 — 네트워크 또는
      원격 저장소 설정 확인 필요"로 표기 후
      사용자에게 계속 진행 여부 확인

   ② git status 로 uncommitted 변경사항 확인
      변경사항 있으면 pull 전 사용자에게
      아래 중 선택 요청:
      a) stash 후 pull
      b) pull 없이 현재 상태 유지
      c) 취소

   ③ 변경사항 없거나 stash 선택 시
      git pull origin {기본 브랜치}
      성공 시 "최신 코드 동기화 완료" 표기
      충돌 발생 시 "pull 충돌 발생 —
      수동 해결 필요" 표기 후 중단.
      충돌 파일 목록 출력.

4. git branch -a
   현재 브랜치 및 전체 브랜치 목록 확인

5. git log --oneline -10
   최근 커밋 이력 확인
   기본 브랜치 기준 미병합 브랜치 존재 여부 확인

6. git status
   미커밋 변경사항 확인
   git stash list 로 스태시 존재 여부 확인

7. 미병합 브랜치가 있으면
   git diff {기본브랜치}..origin/{미병합브랜치} --stat
   로 변경 파일 목록 확인

### Phase 3. 데이터 및 환경 상태 확인

8. 데이터 저장소 현황 파악
   프로젝트에서 사용하는 저장소 유형에 따라
   아래 중 해당 항목 확인:
   - DB 사용 시: 주요 테이블 존재 여부 및 건수
   - 클라우드 스토리지 사용 시: 주요 경로 존재 여부
   - 로컬 파일 기반 시: 주요 파일 존재 여부
   - 해당 없으면 "데이터 저장소 없음"으로 표기

9. 환경 변수 상태 확인
   .env 파일이 존재하면 아래를 확인할 것.
   - 파일 존재 여부
   - 정의된 변수 키 목록
     (실제 값은 절대 출력하지 말 것)
   - .env.example 대비 누락 항목 확인

## 주의사항
- .env 실제 값은 절대 출력하지 말 것
- 데이터 저장소 접근 불가 시
  "접근 불가 — 연결 설정 확인 필요"로 표기
- 미병합 브랜치가 있으면 병합 여부를
  사용자에게 먼저 확인할 것
- 보고 후 추가 지시 없이 자동으로 작업 시작하지 말 것
- 없는 항목은 유연하게 스킵할 것
- 회고 유무에 따라 반드시 분기하여 처리할 것
